Figura 2. Diagrama de Caso de Uso: Registrar usuario (CU-02)
El administrador accede al módulo de gestión de usuarios, ingresa los datos del nuevo
usuario y el sistema valida que no exista duplicado antes de guardar la información.
3. Descripción General
Este caso de uso describe cómo el administrador del sistema registra nuevos usuarios
(estudiantes o administradores) en la plataforma Chill Zone. El objetivo es permitir el
acceso controlado al sistema mediante credenciales únicas.
4. Precondiciones
● El administrador debe haber iniciado sesión.
● El sistema debe estar conectado a la base de datos.
● El correo institucional del nuevo usuario no debe estar registrado previamente.

El estudiante o administrador accede a su historial de reservas, selecciona una reserva

3. Descripción General
Este caso de uso permite a Estudiante/Administrador cancelar una reserva previamente
realizada, siempre que no haya iniciado el uso del recurso. Esto libera el espacio para que
otros usuarios puedan reservarlo.
4. Precondiciones
● El Estudiante/Administrador debe haber iniciado sesión.
● Debe existir una reserva activa que aún no haya comenzado.
● La cancelación debe hacerse con al menos 10 minutos de anticipación.

3. Descripción General
Este caso de uso permite a los usuarios (estudiantes o administradores) reportar cualquier
problema, daño o mal uso detectado en los recursos recreativos del área Chill Zone. El
reporte queda registrado para seguimiento y mantenimiento.

● El Estudiante/Administrador debe haber iniciado sesión.
● El recurso debe estar registrado en el sistema.
● El Estudiante/Administrador debe tener acceso al módulo de incidencias.

1. Identificación del Caso de Uso
Elemento Descripción
Código del Caso de
Uso:
CU-16
Nombre: Generar estadísticas por usuario
Actor principal: Administrador
Actores secundarios: Sistema
Requisitos asociados: RF-05 – El sistema debe generar reportes de uso (por
recurso, usuario, fecha y horario)
Prioridad: Media
Versión: 1.0
2. Diagrama de caso de Uso
Figura 16. Diagrama de Caso de Uso: Generar estadísticas por usuario (CU-16)
El administrador accede al módulo de estadísticas, selecciona un usuario o grupo de
usuarios, y el sistema genera gráficos y datos sobre frecuencia de uso, recursos más
utilizados y horarios preferidos.
3. Descripción General
Este caso de uso permite al administrador generar estadísticas detalladas sobre el
comportamiento de los usuarios en el uso de los recursos recreativos. Esto ayuda a
identificar patrones de uso, usuarios frecuentes y posibles mejoras en la gestión del
espacio.
4. Precondiciones
● El administrador debe haber iniciado sesión.
● Debe existir historial de uso registrado por los usuarios.
● El sistema debe tener acceso a los datos de uso y reservas.
5. Flujo Principal de Eventos
Pas
o
Descripción de la acción Responsable
1 El administrador accede al módulo “Estadísticas”. Administrador
2 Selecciona un usuario o grupo de usuarios. Administrador
3 Define filtros opcionales (rango de fechas, tipo de recurso, etc.). Administrador
4 El sistema consulta el historial de uso del usuario. Sistema
5 El sistema genera estadísticas: número de reservas, recursos
más usados, tiempo promedio de uso, horarios frecuentes.
Sistema
6 Se muestran los resultados. Sistema
6. Flujos Alternativos
Códig
o
Descripción

disponibles”.
FA-02 Si hay error de conexión, se muestra mensaje de reintento.
FA-03 Si los filtros son inválidos, el sistema solicita corrección.
7. Postcondiciones
● Las estadísticas se generan correctamente y están disponibles para análisis.
8. Reglas de Negocio
● Solo el administrador puede acceder a estadísticas de usuarios.
● Los datos deben mantenerse actualizados y reflejar el historial real.
● El sistema debe proteger la privacidad de los usuarios al mostrar datos
agregados cuando sea necesario.

11. Excepciones

● Usuario no encontrado.
● Filtros inválidos o sin resultados.

13. Conclusiones
El caso de uso CU-16 “Generar estadísticas por usuario” permite al administrador analizar
el comportamiento de los estudiantes en el uso de los recursos recreativos, lo que facilita
la toma de decisiones para mejorar la gestión del espacio y promover un uso más
equitativo.

El actor (Estudiante o Administrador) interactúa con el sistema para ingresar sus

3. Descripción General
Este caso de uso describe cómo un estudiante o administrador accede al sistema mediante
credenciales registradas. El objetivo es garantizar el acceso seguro y personalizado a las
funcionalidades del sistema Chill Zone.

Prioridad: Media
Versión: 1.0
2. Diagrama de caso de Uso
Figura 3. Diagrama de Caso de Uso: Recuperar contraseña (CU-03)
El estudiante o administrador solicita la recuperación de su contraseña. El sistema valida
el correo institucional y envía un enlace temporal para restablecer la contraseña..
3. Descripción General
Este caso de uso permite a los estudiantes recuperar el acceso al sistema en caso de olvidar
su contraseña. El sistema enviará un enlace de recuperación al correo institucional
registrado.
4. Precondiciones
● El estudiante debe estar registrado en el sistema.
● El correo institucional debe estar activo y accesible.
5. Flujo Principal de Eventos
Pas
o
Descripción de la acción Responsabl

or

Administrad
or
3 El sistema valida que el correo exista en la base de datos. Sistema
4 El sistema genera un token temporal y lo envía por correo con
un enlace de recuperación.
Sistema
5 El Estudiante/Administrador accede al enlace, ingresa una
nueva contraseña y la confirma.
Estudiante/
Administrad

éxito.
Sistema
6. Flujos Alternativos
Código Descripción
FA-01 Si el correo no está registrado, el sistema muestra: “Correo no encontrado”.
FA-02 Si el enlace ha expirado, el sistema solicita generar uno nuevo.
FA-03 Si hay error de conexión, se muestra un mensaje de reintento.
7. Postcondiciones
● La contraseña del usuario se actualiza correctamente.
8. Reglas de Negocio
● El enlace de recuperación tiene una validez de 15 minutos.
● La nueva contraseña debe cumplir con los requisitos de seguridad (mínimo 8
caracteres, al menos una mayúscula, un número y un símbolo).
● Solo se permite un intento de recuperación cada 5 minutos.

2. Diagrama de caso de Uso
Figura 5. Diagrama de Caso de Uso: Consultar disponibilidad (CU-05)
El estudiante o administrador accede al módulo de reservas y consulta en tiempo real la
disponibilidad de los recursos recreativos (mesas, televisores, futbolines, etc.) por fecha
y franja horaria.
3. Descripción General
Este caso de uso permite a los estudiantes visualizar la disponibilidad actual de los
recursos recreativos del área Chill Zone antes de realizar una reserva. La información se
presenta de forma clara y actualizada.
4. Precondiciones
● El estudiante debe haber iniciado sesión.
● El sistema debe tener acceso a la base de datos de reservas y estado de los
recursos.

El estudiante o administradorconsulta la disponibilidad de un recurso, selecciona fecha y
hora, y realiza la reserva. El sistema valida la disponibilidad y envía una confirmación.
3. Descripción General
Este caso de uso permite a los estudiantes reservar recursos recreativos disponibles en el
área Chill Zone (mesas, televisores, futbolines, etc.) por franjas horarias. La reserva queda
registrada en el sistema y se notifica al usuario.
4. Precondiciones
● El estudiante debe haber iniciado sesión.
● El recurso debe estar disponible en la fecha y hora seleccionada.
● No debe haber otra reserva activa del mismo usuario en ese horario.

6. Flujos Alternativos
Código Descripción
FA-01 Si el recurso ya está reservado, el sistema muestra: “No disponible en ese
horario”.
FA-02 Si el Estudiante/Administrador tiene otra reserva activa en el mismo
horario, se muestra advertencia.
FA-03 Si hay error de conexión se notifica al Estudiante/Administrador.
7. Postcondiciones
● La reserva queda registrada en la base de datos.
● El Estudiante/Administrador recibe confirmación de la reserva.
● El recurso queda marcado como reservado para ese horario.
8. Reglas de Negocio
● Un Estudiante/Administrador no puede tener más de una reserva activa en el
mismo horario.
● Las reservas deben realizarse con al menos 15 minutos de anticipación.
● El sistema debe bloquear automáticamente los recursos reservados o en
mantenimiento.
